# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Backend (Flask REST API + Socket.IO) for **Portfolio-MDE Dashboard** — a management dashboard for a *Maison De l'Entreprise* (business incubator). It manages users, hosted companies, trainings/sessions, news, notifications, and a real-time chat. The React/Tailwind frontend lives in `frontend/` (separate Vite app).

## Commands

All backend commands run from the repo root with the venv active (`source .venv/bin/activate`). Code uses absolute imports rooted at `backend.*`, so **always run from the repo root** (or `PYTHONPATH=.`), never from inside `backend/`.

```bash
# Run the dev server (Socket.IO-aware entrypoint) on :8000
python -m backend.api.run

# Tests (pytest.ini sets testpaths=tests)
pytest                                   # full suite
pytest tests/test_facades.py             # one file
pytest tests/test_facades.py::test_name  # one test
pytest -k "conversation"                 # by keyword

# Create/reset the default super admin (admin@admin.com / admin)
python create_super_admin.py

# DB migrations (Alembic; config in alembic.ini, env in alembic/)
alembic upgrade head
alembic revision -m "description"

# Frontend (from frontend/)
npm install && npm run dev               # Vite dev server on :3000
```

API docs: Swagger UI is served at `/`, `/docs`, `/swagger`; raw spec at `/openapi.json`. Health check at `/status`.

## Architecture

Strict layering — each layer only talks to the one below it:

```
HTTP request
  → Resource           backend/api/resources/*.py     (flask_restful, routing, auth)
  → Service            backend/persistence/services/layer.py   (thin business wrappers)
  → Facade             backend/persistence/services/facades/*_sql.py  (all SQLAlchemy/CRUD)
  → ORM models         backend/persistence/models.py  (SQLAlchemy tables)
```

Key conventions when adding or changing features:

- **Two model worlds, do not confuse them.** `backend/models/*.py` are plain (non-ORM) *domain* models whose job is **validation** via property setters (see `models/user.py`). `backend/persistence/models.py` are the *SQLAlchemy* tables. Resources validate input by instantiating the domain model (e.g. `DomainUser(email=..., password=...)` raises `ValueError`/`TypeError`), then persist via the service. Password hashing is deliberately NOT in the domain model — it happens in `user_facade_sql.py` (werkzeug `generate_password_hash`).
- **Facades own all DB access.** Every facade method opens a transaction via `session_scope()` from `facades/_common_sql.py` (commit on success, rollback on exception). Facades return **dicts**, not ORM objects (sessions are closed on return). Reuse `_common_sql.py` helpers: `to_csv`/`from_csv`, `isoformat`, `normalize_text`.
- **Services are intentionally thin.** Most services in `layer.py` just hold `self.facade`; resources/socket handlers often call `service.facade.<method>` directly. Import services from `backend.persistence.services` (re-exported in its `__init__.py`).
- **IDs are string UUIDs** (`String(36)`), generated by `default=lambda: str(uuid.uuid4())`. Soft-delete pattern: rows carry `is_active`, `deactivate_by`, `delete_by` rather than being hard-deleted.

### Auth

JWT via `flask_jwt_extended`. Use the `jwt_required` wrapper from `backend/api/jwt_helpers.py` (not the library's directly). Logout is implemented as an **in-memory** JTI blocklist in `backend/api/state.py` (`BLOCKLIST`) — it does not survive a restart, and is shared by both REST (`token_in_blocklist_loader`) and sockets. `JWT_SECRET_KEY` comes from the env (a weak default is hardcoded in `app.py`). All JWT error shapes are normalized through `backend/api/errors.py` → `error_response(ERROR_CODES[...], msg, status)`; follow that pattern for new error returns rather than raw `abort`.

### Real-time chat (Socket.IO)

- Single `SocketIO` instance lives in `backend/api/sockets.py`; events are in `backend/api/socket_events/message.py`; `app.py` does `socketio.init_app(app)`.
- Socket auth is separate from REST: clients pass `{ token }` in the connection `auth`; `socket_events/message.py:handle_connect` verifies it with `backend/api/socket_auth.py:verify_token` and maps `sid → user_id` in the module-level `connected_users` dict.
- Rooms are namespaced `conversation:<id>` (`conversation_room()`). Events: `join_conversation`, `leave_conversation`, `send_message` → server emits `new_message`/`joined_conversation`/`left_conversation`.
- **Run the server via `backend.api.run` / `socketio.run`, not `app.run`**, or websockets won't work. See `CHAT_GUIDE.md` for the full event contract.

### Database lifecycle (important gotcha)

`backend/persistence/db.py` does a lot of best-effort schema work *at import time*: it ensures the SQLite file exists, runs `create_all()`, and even **drops & recreates all tables** if the `conversation_participants` table is missing an expected column. Alembic migrations exist in `alembic/versions/` but the runtime also self-heals the schema. The dev DB is `data.db` in the repo root (`DATABASE_URL` env overrides; `psycopg2-binary` is installed for Postgres in prod). Tests bypass all this: `tests/conftest.py` points `DATABASE_URL` at a fresh `tmp_path` SQLite file, clears all `backend.*` modules from `sys.modules` so the engine is rebuilt per test, and `drop_all`/`create_all` for a clean schema. Use the `app_bundle` and `seeded_context` fixtures (seeds a super admin + company admin + member with login headers).

### Background jobs

`app.py` starts an APScheduler `BackgroundScheduler` that runs `backend/services/news_sync.py:sync_all` hourly (fetches French business-news RSS feeds via `feedparser`). It is skipped when `app.testing` and guarded against the Werkzeug reloader double-start. `backend/services/` (note: top-level, distinct from `backend/persistence/services/`) holds these cross-cutting jobs.

The **CSV column is the source of truth**; the `conversation_participants` table is currently dormant (kept for a future migration). Be deliberate about which one you read/write.
